# Claude Code 约束文件生成 Skill

帮助用户为项目生成一份专属的 `CLAUDE.md`，让 Claude Code 在执行前、执行中、执行后都在可控范围内工作。

## 核心思路

**不要直接生成通用模板**——先读懂项目，再生成专属约束。
通用模板里的路径和红线都是假的占位符，没有实际约束力。

---

## 第一步：分析项目结构

在生成 CLAUDE.md 之前，先对项目做全面分析。运行以下命令（根据实际情况调整）：

```bash
# 查看目录结构
find . -type f | grep -v node_modules | grep -v .git | grep -v dist | head -60

# 查看依赖
cat package.json 2>/dev/null || cat requirements.txt 2>/dev/null || cat go.mod 2>/dev/null

# 查看现有测试
find . -name "*.test.*" -o -name "*.spec.*" | grep -v node_modules | head -20

# 查看现有配置文件
ls -la | grep -E "\.(env|config|json|yaml|yml|toml)$"
```

分析后，整理出以下信息，**输出给用户确认**，不要直接开始生成文件：

1. **目录结构**：各模块的名称和职责
2. **技术栈**：语言、框架、主要依赖
3. **核心模块**：哪些目录/文件是不能轻易改动的
4. **共享模块**：公共组件、工具函数等
5. **测试框架**：用了什么测试工具，测试文件在哪里
6. **代码风格**：有没有 ESLint/Prettier/.editorconfig 等规范文件

等用户确认分析结果无误后，再进入第二步。

---

## 第二步：生成 CLAUDE.md

基于第一步确认的分析结果，生成以下结构的 `CLAUDE.md`，放在**项目根目录**。

## 🚫 红线规则（禁止行为）

以下规则**不可违反**，遇到冲突时以本规则为准：

- 禁止修改 `<核心模块路径>` 目录下的任何文件
- 禁止修改数据库 schema / migration 文件
- 禁止删除或重命名已有的公共接口和方法
- 禁止引入新的第三方依赖，除非用户明确同意
- 禁止修改配置文件（如 .env、<config文件>）
- 禁止修改 CI/CD 配置文件

## ✅ 执行前（每次任务开始前必须做）

1. 列出本次任务**涉及的所有文件**（新增 / 修改 / 删除分别列出）
2. 说明**实现方案和思路**，包括关键技术决策
3. 标明是否涉及**共享模块或核心模块**
4. **等待用户确认后**再开始写代码，不要假设用户已同意

---

## ⚙️ 执行中（过程规范）

- 每完成一个**独立的小步骤**后暂停，等待用户确认再继续
- 遇到方案有多种选择时，**列出选项让用户决策**，不要自行决定
- 发现已有代码有问题时，**告知用户但不要顺手修改**
- 只改和本次需求**直接相关**的代码，不做范围外的"顺手优化"
- 不确定的地方**主动问**，不要做假设后默默执行

---

## ✅ 执行后（每次任务完成后必须做）

1. 为新功能补充**单元测试**（正常路径 + 边界值 + 异常情况）
2. 列出**所有被修改的文件清单**
3. 说明是否有**潜在副作用**或需要注意的地方
4. 告知用户需要**手动验证的点**

## 第三步：让 Claude Code 自我验证

生成 CLAUDE.md 后，追加以下提示让它验证：

```
请重新读一遍你生成的 CLAUDE.md，检查：
1. 所有目录路径是否和项目实际结构一致
2. 禁止修改的模块名称是否真实存在
3. 测试框架和配置文件名称是否正确
如有不符请直接修正。
```

## 使用建议

### CLAUDE.md 持续进化
每次出现问题，就在「历史教训」部分补充一条记录：
```
- 2026-06 修改了 shared/auth.ts 导致登录异常
  → 今后修改 shared/ 下任何文件前必须单独告知用户
```

### 配合 git 使用
```bash
# 每个小步骤完成后提交，方便追溯和回滚
git add -p          # 逐块审查，不要 git add .
git commit -m "step1: 说明做了什么"
```

### 任务拆分原则
不要一次性把复杂需求全部交给 Claude Code，拆成小块：
- 第一步：数据层
- 第二步：业务逻辑层（确认数据层后再开始）
- 第三步：UI 层

每步完成后 `git diff` 看改动，确认无误再继续。
