# 前端部署与缓存机制问答

> 围绕「SPA 部署后少量用户白屏」问题,展开的 CDN 与 HTTP 缓存机制问答整理。

---

## Q1:部署后少部分用户白屏是什么问题?怎么全面解决?

**背景:** React 打包后交给运维部署到 Cloudflare + CDN,运维用新包替换旧包。每次部署后,2 万用户里有几百个反馈页面空白,大部分用户正常。

**根本原因:** SPA 的「资源版本断裂」。`index.html` 里写死了对某一批带 hash 资源的引用,而 `index.html` 和 `assets/*` 的更新不是原子的,加上 CDN 缓存被切割,导致新旧版本资源混用。

白屏用户通常命中以下情况之一:

1. **新旧版本资源混用(最常见)**
   运维用新包替换旧包时,旧的 `index.a1b2c3.js` 被删掉了。老用户浏览器里打开的还是旧页面,触发懒加载去请求旧 chunk → 文件已不存在 → 404 → `ChunkLoadError` → 白屏。

2. **CDN 边缘节点缓存不一致**
   CDN 有几百个边缘节点,不会同时刷新。某节点已更新为新 html(引用新 hash),但新 assets 还没同步 → 拿到新 html 却加载不到对应 js → 白屏。

3. **`index.html` 被错误地长缓存**
   用户长期拿到指向已删除资源的旧入口。

**为什么是「少部分」用户:** 出问题需同时满足「在部署时间窗口内访问」+「命中未同步的边缘节点」+「触发缺失的 chunk」,概率窗口很小,所以随机、难复现、部署一段时间后自动消失。

**全面解决方案(按性价比排序):**

| 方案                     | 做法                                                 | 作用                                 |
| ------------------------ | ---------------------------------------------------- | ------------------------------------ |
| 一、资源只增不删(最关键) | 新包**追加**而非覆盖删除,保留最近 3-5 个版本         | 旧 chunk 依然可加载,消灭绝大部分白屏 |
| 二、正确缓存策略         | index.html 不长缓存 + hash 资源永久缓存              | 入口永远最新,资源高效复用            |
| 三、部署顺序原子化       | 先传资源 → 确认同步 → 最后切 index.html → purge 缓存 | 避免「html 指向尚未就绪的资源」      |
| 四、前端兜底             | 捕获 `ChunkLoadError` 后自动刷新一次(带防死循环标记) | 用户无感自愈,不白屏                  |
| 五、版本感知             | 轮询 `version.json`,发现新版本提示用户刷新           | 长会话用户不长期用旧资源             |

**最优先落地:** 方案一 + 方案二 + 方案四,能解决 90%+。长期最优是迁移到 **Cloudflare Pages**(原子部署 + 自动版本管理 + 自动回滚)。

## Q2:「index.html 不长缓存 + hash 资源永久缓存」是什么意思?

构建产物天然分两类,特性完全相反:

### 第一类:带 hash 的资源(`assets/*.[hash].js/css`)

文件名里的 hash 是**内容指纹**:

> 内容不变 → hash 不变 → 文件名不变;内容变了 → hash 变 → 文件名也变。

同名文件的内容永不改变,所以应**永久缓存**:

```
Cache-Control: public, max-age=31536000, immutable
```

- `max-age=31536000` = 缓存 1 年
- `immutable` = 文件不会变,连「问一下有没有更新」都不用

### 第二类:入口文件 `index.html`

文件名固定(不带 hash),但**内容每次发版都变**(引用的 hash 文件名变了)。它是应用的「入口地图」。若长缓存,用户会拿到指向已删除资源的旧 html → 白屏。所以:

```
Cache-Control: no-cache
```

含义不是「不缓存」,而是「可缓存,但每次用前必须向服务器校验」。

### 总结表

| 文件类型                 | 文件名  | 内容会变吗   | 缓存策略                              | 含义              |
| ------------------------ | ------- | ------------ | ------------------------------------- | ----------------- |
| `assets/*.[hash].js/css` | 带 hash | 同名永不变   | `public, max-age=31536000, immutable` | 永久缓存,不再询问 |
| `index.html`             | 固定名  | 每次发版都变 | `no-cache`                            | 每次用前必须校验  |

**核心逻辑:** 入口每次校验保证「永远拿到正确的地图」,hash 资源永久缓存保证「性能和复用」。

### Cloudflare `_headers` 配置示例

```
/
  Cache-Control: no-cache
/index.html
  Cache-Control: no-cache

/assets/*
  Cache-Control: public, max-age=31536000, immutable
```

## Q3:`no-cache`、`no-store`、`must-revalidate` 有什么区别?各适合什么场景?

心智模型:缓存有两个动作——**存不存**、**用之前验不验**。

### `no-store` —— 完全不缓存

不保存任何副本,每次全量重新下载。最严格。
**适合:** 含敏感信息的响应(银行账户页、含 token 的接口、支付页、余额/交易 API)。

### `no-cache` —— 可以存,但用前必须验证(名字骗人)

其实**会缓存**,只是每次用前强制去服务器校验(ETag/Last-Modified)。没变返回 304(省流量),变了返回 200。
**适合:** `index.html`、频繁更新但不敏感的资源。

### `must-revalidate` —— 过期后才验证(不同维度)

通常与 `max-age` 一起用,管的是「过期之后怎么办」:有效期内直接用,过期后**必须**验证,不许用过期旧副本。
**适合:** 有时效但过期后绝不能用旧数据的场景(价格、汇率、余额)。

### 对比表

| 指令              | 存本地吗 | 用前验证吗       | 一句话                        |
| ----------------- | -------- | ---------------- | ----------------------------- |
| `no-store`        | 不存     | 无缓存可用       | 每次全量重新下载              |
| `no-cache`        | 存       | **每次**都验证   | 存了但每次用前必问服务器      |
| `must-revalidate` | 存       | **过期后**才验证 | 有效期内直接用,过期后必须验证 |

**关键辨析:**

- `no-cache` vs `must-revalidate`:前者**每次**都验,后者**只在过期后**验。
- `no-store` vs `no-cache`:前者真不存(敏感数据),后者存了但每次验(入口文件)。

### 部署场景选择

| 文件                     | 推荐配置                              |
| ------------------------ | ------------------------------------- |
| `index.html`             | `no-cache`                            |
| `assets/*.[hash].js/css` | `public, max-age=31536000, immutable` |
| 敏感 API(余额/交易)      | `no-store`                            |
| 有时效的数据接口         | `max-age=N, must-revalidate`          |

`immutable` 是 `no-cache` 的反面:文件永远不会变,过期前连验证都不用。

## Q4:为什么 hash 资源要永久缓存不再询问?每次打包后不是变了吗?

关键点:**不是「同一个文件变了」,而是「变成了一个全新的文件(新文件名)」。**

hash = 内容指纹:

- 文件代码没改 → hash 一样 → 文件名不变,内容一模一样
- 文件代码改了 → hash 变 → 生成**新名字**的文件,和旧的是两个不同文件

**同一个文件名的内容永远不会变**,代码改动产生的是「新文件名」而非「覆盖旧文件名」。

### 发版例子

发版前:

```
index.a1b2c3.js     ← 业务代码
vendor.d4e5f6.js    ← 第三方库
```

只改业务代码,重新打包后:

```
index.x9y8z7.js     ← 业务代码变了,hash 变 → 新文件名
vendor.d4e5f6.js    ← 第三方库没改,hash 没变 → 文件名不变!
```

- `vendor.d4e5f6.js` 文件名一样 → 浏览器**直接复用缓存,一字节都不下载**
- 业务代码变成新名字 → 只下载这一个新文件

**为什么能永久缓存不询问:** 既然规则保证「这个文件名的内容永不变」,浏览器就没理由再问服务器「更新了吗」——它不可能更新,更新的话就叫别的名字了。

**新版本怎么被拿到:** 靠 `index.html` 这张「每次都更新的地图」指向新的 hash 文件。

| 角色                 | 谁                   | 缓存策略             | 职责                          |
| -------------------- | -------------------- | -------------------- | ----------------------------- |
| 地图(会变,不带 hash) | `index.html`         | `no-cache` 每次校验  | 告诉浏览器加载哪些文件        |
| 内容(名字=指纹)      | `assets/*.[hash].js` | `immutable` 永久缓存 | 名字没变就复用,变了就是新文件 |

这套设计叫 **cache busting**:用文件名变化「绕过」缓存,而不是靠让缓存过期。

## Q5:这些缓存配置是浏览器获取资源前读取来决定读本地还是向服务器申请的吗?

方向正确,但需精确一个时间点:

> `Cache-Control` **不是浏览器凭空就有的**,而是**服务器在上一次响应时,跟着文件一起下发、并存在本地的**。

**流程:**

第一次请求(本地没有):

1. 向服务器请求
2. 服务器返回**文件内容** + 响应头 `Cache-Control: ...`
3. 浏览器把**文件内容和缓存规则一起存到本地**

第二次需要时:

1. 查本地缓存,读**当初一起存下的规则**
2. 判断:直接用 / 验证后用 / 重新下载

### 三种结果

| 情况                      | 浏览器行为          | 网络请求                           |
| ------------------------- | ------------------- | ---------------------------------- |
| 有效期内(如 immutable)    | 直接用本地缓存      | **无请求**(显示 disk/memory cache) |
| 需验证(no-cache 或已过期) | 发校验请求(带 ETag) | 有,可能返回 304                    |
| 本地没有 / no-store       | 发完整请求下载      | 有完整请求                         |

### 重要补充:CDN 也读这些规则

架构是 `浏览器 → Cloudflare CDN → 源站`,`Cache-Control` **每一层都遵守**。CDN 边缘节点也会缓存,各节点同步有时间差——这正是「CDN 缓存不一致导致白屏」的原因。

**修正后的心智模型:** 缓存规则来自上一次响应,和文件一起缓存在本地;浏览器(和 CDN)下次用前读这条本地规则,决定「直接用 / 问一下再用 / 重新下载」。

## Q6:index.html 加 no-cache 后,每次验证它是最新的——怎么判断新不新?

靠服务器给文件打的两种「身份标记」:**ETag** 和 **Last-Modified**。

### 服务器打标记

第一次返回时附带:

```
ETag: "a3f5b2c8"                              ← 内容指纹(哈希)
Last-Modified: Tue, 08 Jul 2026 10:00:00 GMT  ← 最后修改时间
```

浏览器把 html 和这两个标记一起存本地。

### 验证过程

下次访问带上标记去问:

```
If-None-Match: "a3f5b2c8"
If-Modified-Since: Tue, 08 Jul 2026 10:00:00 GMT
```

服务器比对当前文件:

- **没变:** ETag 一致 → 返回 `304 Not Modified`(空 body,几十字节)→ 用本地缓存
- **变了:** ETag 不一致 → 返回 `200 OK` + 新 ETag + 新内容 → 用新的并更新缓存

### ETag vs Last-Modified

|               | 依据     | 精度       | 问题                                               |
| ------------- | -------- | ---------- | -------------------------------------------------- |
| Last-Modified | 修改时间 | 秒级       | 时间只到秒;重新保存但内容没变会误判;多机时间不一致 |
| ETag          | 内容指纹 | 精确到内容 | 更可靠                                             |

ETag 更准,主流以 ETag 为主,两者都存在时 ETag 优先。

**部署提醒:** ETag 通常由服务器/CDN 自动生成,一般不用手动设。但**多台源站**时,不同机器对同一文件生成的 ETag 可能不同(尤其 nginx 基于 inode 的默认 ETag),会导致没变却总返回 200 重新下载——需统一 ETag 生成规则。

## Q7:`public, max-age=0, must-revalidate` 是什么意思?

逐个拆解:

- **`public`** — 任何缓存(浏览器、CDN、代理)都可以存。对应 `private`(仅浏览器)。
- **`max-age=0`** — 新鲜期 0 秒,一存下就立刻过期。
- **`must-revalidate`** — 一旦过期必须去服务器验证,验证前不许用旧的。

**组合效果:** 每次用前都算过期 → 每次都必须验证 → 带 ETag 去问,没变 304 用本地,变了 200 拿新的。**效果与 `no-cache` 几乎等价。**

### 与 no-cache 的区别

|        | `no-cache`               | `max-age=0, must-revalidate`            |
| ------ | ------------------------ | --------------------------------------- |
| 语义   | 直接表达「用前必须验证」 | 用「立即过期 + 过期必须验证」拼出同效果 |
| 兼容性 | 现代浏览器都支持         | 更老更保险,老旧代理处理更稳             |
| 来源   | 手写常用                 | 框架/服务器/CDN 常见默认值              |

**少了 `must-revalidate` 的坑:** 只写 `max-age=0` 时,部分缓存/代理在「过期 + 连不上服务器」时可能将就用过期旧副本。加上 `must-revalidate` 明确禁止——宁可报错也不用旧数据。对 index.html 正是所需。

**结论:** 用在 `index.html` 上完全正确合适,不用改。要检查的是 `assets/*.[hash].js/css` 是否配成了 `max-age=31536000, immutable`。

## Q8:这样配置 index.html 是不是就没有缓存了?

**不是。它仍然有缓存,只是每次用前都要验证。**「有缓存但每次验证」和「没有缓存」是两码事。

真正禁止存储的是 `no-store`。

| 配置                                      | 存本地吗 | 用之前            |
| ----------------------------------------- | -------- | ----------------- |
| `no-store`                                | **不存** | 每次完整重新下载  |
| `max-age=0, must-revalidate` / `no-cache` | **存!**  | 每次验证(带 ETag) |

### 「存了但每次验证」省了什么

验证通过时**不用重新下载文件内容**:

- `no-store`:每次都传完整 html(如 5KB)
- `no-cache`:没变时只传几十字节的 304,本地 5KB 直接复用

### 三种状态一句话概括

- **完全没缓存** = `no-store` → 每次全量下载
- **有缓存但每次验证** = `no-cache` / `max-age=0, must-revalidate` → 存本地,每次问,没变用本地(304)
- **有缓存且一段时间内不验证** = `max-age=3600` → 有效期内直接用

index.html 属于中间那种。**不用 `no-store` 的原因:** 会浪费掉 304 优化(两次发版之间 html 通常没变),`no-cache` 既保证最新又省流量。

## Q9:`Cf-Cache-Status: DYNAMIC` 是什么意思?

Cloudflare 专有响应头,表示这次请求**在 CDN 边缘节点有没有命中缓存**。

### 常见值

| 值            | 含义                                           |
| ------------- | ---------------------------------------------- |
| `HIT`         | 命中缓存,边缘节点直接返回,没回源(最理想)       |
| `MISS`        | 边缘没有,回源取并按规则缓存,下次可能 HIT       |
| `EXPIRED`     | 缓存过期,回源重取                              |
| `REVALIDATED` | 过期后验证发现没变,继续用                      |
| **`DYNAMIC`** | Cloudflare 认为不可缓存,每次回源,没进 CDN 缓存 |
| `BYPASS`      | 有规则明确跳过缓存                             |

### DYNAMIC 的含义

响应**没被 Cloudflare 缓存**,每次请求都穿透 CDN 回源站。Cloudflare 默认只缓存它认为是静态资源的文件(靠扩展名),而 HTML、接口、无明确规则的路径默认不缓存,标成 `DYNAMIC`。

### 好坏取决于是哪个文件

- **`index.html` 是 DYNAMIC:** 基本没问题,符合预期(本就不该长缓存)。优化点:让 Cloudflare 做 `no-cache` 式缓存,帮源站挡掉部分完整传输。
- **`assets/*.js/css` 是 DYNAMIC:** ⚠️ 有问题!hash 资源本该 `HIT`,却每次回源 → 源站压力大、加载慢、失去 CDN 核心价值。

### hash 资源变 DYNAMIC 的排查方向

1. 源站响应头带了 `private` / `no-store` / `set-cookie` 等阻止缓存的指令
2. URL 带查询参数且未配置忽略
3. 没配 Cache Rule / Page Rule 显式要求缓存
4. 路径/扩展名不在默认缓存列表

### 验证动作

```bash
# index.html —— DYNAMIC 可接受
curl -sI https://你的域名/index.html | grep -i cf-cache-status

# hash 资源 —— 期望 HIT,绝不该 DYNAMIC
curl -sI https://你的域名/assets/index.xxxxxx.js | grep -i cf-cache-status
```

| 文件                     | 期望                      | 若为 DYNAMIC   |
| ------------------------ | ------------------------- | -------------- |
| `index.html`             | `DYNAMIC` / `REVALIDATED` | 可接受         |
| `assets/*.[hash].js/css` | `HIT`(预热后)             | ❌ 有问题,要修 |

### 与白屏问题的关联

若 hash 资源是 `DYNAMIC`(每次回源),运维删旧文件的瞬间,用户请求旧 chunk 直接穿透到源站拿 404 → 白屏。若 CDN 把旧 hash 资源缓存成 `HIT`,源站删文件后边缘节点还留旧副本,能多扛一会儿,减少白屏。

**一句话:** `Cf-Cache-Status: DYNAMIC` = 这次没走 CDN 缓存、直接回源。出现在 index.html 正常;出现在 hash 资源上说明 CDN 没缓存静态资源,需检查配置。
